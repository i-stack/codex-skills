# 根因修复铁律（Root Cause Enforcement）

所有问题排查与修复**必须**遵循以下强制规则，任何实现不得违反。

---

## 7.1 根因修复是必选项

- 根因修复是**强制要求**，不是优先选项
- 所有问题必须先定位根因，再实施修复
- 禁止绕过根因直接做表层处理
- 若无法确认根因，**不得实施修复代码**

## 7.2 禁止补丁式修复

禁止通过以下方式掩盖问题：

- 兜底 `if` 判断
- 延迟回调（`DispatchQueue.asyncAfter`）
- 重复刷新（`reloadData` / `setNeedsLayout`）
- 临时状态变量
- 多余容错逻辑
- 防御式分支掩盖异常

修复必须解决**产生错误的真实结构原因**，而不是掩盖现象。

## 7.3 禁止主观猜测定位

不得凭经验或直觉猜测问题位置，必须基于以下证据进行定位：

- 调用链路分析
- 状态流转追踪
- 生命周期推演
- 并发域隔离检查（线程 / Actor / Task）
- 内存引用关系验证
- 调用栈与日志证据

**没有证据，不得下结论。**

## 7.4 必须进行全链路排查

问题排查必须沿完整技术链路进行：

```
数据源 → 状态管理 → 并发模型 → 生命周期 → 渲染路径 → 事件响应
```

禁止仅在 View 层或局部组件层做修补。

## 7.5 必须结构性修复

修复应优先落在以下层级：

- 架构层
- 状态模型层
- 数据流层
- 并发边界层

**禁止**通过以下方式解决：

- 在 View 层追加分支
- 增加临时标记位
- 增加 UI 状态兜底

## 7.6 必须具备可观测性

在排查和修复过程中，关键节点必须输出日志或埋点。

日志必须覆盖：

| 类别 | 说明 |
|------|------|
| 输入数据 | 方法入参、外部输入 |
| 状态变化 | 状态机流转、属性变更 |
| 线程/Actor | 当前执行上下文 |
| 生命周期 | init/deinit/viewDidLoad 等 |
| UI 更新触发点 | 刷新来源与时机 |
| 异常分支 | guard/catch/else 路径 |

日志质量要求：**可复现、可追踪、可对比**。

## 7.7 必须先定义问题边界

在开始排查前，必须明确：

- 触发条件
- 影响范围
- 是否必现或概率触发
- 相关设备/系统版本
- 线程或并发环境

**未定义问题边界，不得进入修复阶段。**

## 7.8 必须区分"现象"与"根因"

- 崩溃点或异常日志通常只是**表象**
- 不得将最终报错位置直接等同于根因
- 必须继续向上追踪触发源头

## 7.9 必须评估修复副作用

任何修复都必须评估是否影响：

- 状态流
- 并发安全
- 性能
- 内存释放
- UI 复用链路
- 其他业务路径

**禁止以解决一个问题为代价引入新的隐性问题。**

## 7.10 修复必须可验证

修复必须通过以下方式进行验证：

- 单元测试
- 集成测试
- 日志断点
- Instruments
- 内存图
- 并发检查工具

**没有验证证据，不得认为问题已修复。**

## 7.11 必须沉淀排查资产

对于典型问题，必须保留：

- 最小复现方式
- 关键日志样例
- 根因说明
- 修复策略

确保团队可以复用排查经验，避免重复踩坑。

## 7.12 必须形成闭环结论

问题修复必须形成完整闭环：

```
问题现象 → 触发条件 → 根因分析 → 修复方案 → 验证结果
```

缺少任何一项，视为**未完成问题修复**。

---

## 总结

| 原则 | 说明 |
|------|------|
| 没有证据，不得下结论 | 禁止主观猜测 |
| 没有复现，不得称已定位 | 必须可复现 |
| 没有验证，不得称已修复 | 必须有验证证据 |
| 禁止补丁式修复 | 禁止掩盖现象 |
| 修复必须落在根因层 | 结构性修复 |
